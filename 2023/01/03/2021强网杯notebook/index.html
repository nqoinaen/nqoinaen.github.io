2021强网杯notebook

本文最后更新于：2023年1月13日 下午

前言

每一道kernel题感觉都学到好多

0x01分析

1.保护

开启了kaslr，smep/smap (smep用户代码不可执行、smap用户数据不可访问)

开启了KPTI(Kernel page-table isolation内核页表隔离)

cat /sys/devices/system/cpu/vulnerabilities/*

2.notebook.ko

惯例看漏洞驱动

一.ioctl

菜单选项,add,gift,del,edit

add向slub申请不大于0x60的object，

有个copy_from_user拷贝用户数据到内核全局变量name,而这个name仅用作于打印，可以认为出题人放在这里是给我们userfaultfd的，虽然加了读锁，不过读锁是可以并发的

del加了写锁，也就是独占锁（可以用edit的krealloc来达到kfree的作用，edit里没有写锁）

edit用krealloc来改变object的大小，这里edit的newsize没有限制，可以任意大小

krealloc的特性：

krealloc的new_size=NULL，则kfree释放object

krealloc的new_size<ks，则指针不变，仅调整kasan监控的区域

krealloc的new_size>ks,则kfree释放原本object,kmalloc重新申请新object

这里也有个多余的copy_from_user到name

gift把申请object的地址传给用户

二.read&write

两者都用了_check_object_size来检查size,防止了溢出

read就是读object指定size字节给用户

write就是从用户地址指定size字节给object

0x02解法1：

1.由add和edit配合申请0x10个TTY_STRUCT_SIZE(0x2e0)的object,然后都去edit一个更大的size，借由krealloc的特性以达到kfree，

然后在下面copy_from_user用的是注册userfaultfd的buf,都使其暂停，达到了uaf的效果

2.进行heap spray,多次重复打开/dev/ptmx，以求大概率分配某个释放的object为 tty_struct

此外在 read 和 write 中都会用 _check_object_size 检查 size 与 buf 大小是否匹配，在 mynote_add 当中限制了 size 应当不大于 0x60，而我们在 edit 中的释放操作之前会将 size 改掉，考虑到在 mynote_add 中先用 copy_from_user 拷贝数据后才调用 kmalloc，故这里还是可以新开 add 线程让 size 合法后通过 userfaultfd 让其卡在这里

3.借由gift给的地址，检查magic是否为0x5401来判断是否分配到tty_struct

4.通过tty结构中的 tty_operations泄露内核地址

ptm_unix98_ops && pty_unix98_ops

在 ptmx 被打开时内核通过 alloc_tty_struct() 分配 tty_struct 的内存空间，之后会将 tty_operations 初始化为全局变量 ptm_unix98_ops 或 pty_unix98_ops，在调试阶段我们可以先关掉 kaslr 开 root 从 /proc/kallsyms 中读取其偏移

开启了 kaslr 的内核在内存中的偏移依然以内存页为粒度，故我们可以通过比对 tty_operations 地址的低三16进制位来判断是 ptm_unix98_ops 还是 pty_unix98_ops

5.劫持tty_ops的write，通过程序运行那时rdi寄存器存的tty_struct地址，找gadget三次栈迁移

从tty_ops->write到tty_struct到tty_ops到notes

在notes写入rop链，用commit_creds(init_cred)即可，返回用户态用swapgs_restore_regs_and_return_to_usermode，顺便把KPTI过了

（其实我感觉只要两次栈迁移即可，直接tty_struct到notes，但是我尝试多次无果）

5.最后遍历write某个分配到tty_struct的设备来触发即可

Exp

因为已经懒得一步步写了，总之就是在理解的基础上(自认为)各种cv

其中设置cpu和对线程的顺序访问，还有在创建线程后sleep一会，都能提高一下稳定性应该

cpu_set_t pwn_cpu;
CPU_ZERO(&pwn_cpu);
CPU_SET(0, &pwn_cpu);

sem_init(&sem_add, 0, 0);
sem_post(&sem_edit);
sem_wait(&sem_edit);

sleep(1)

但是实际这种打法还是不太稳定，长亭有种解法用work_for_cpu_fn函数,能稳定利用

0x03解法2：

就是用work_for_cpu_fn这个函数，可以不用rop

该函数位于 workqueue 机制的实现中，只要是开启了多核支持的内核 （CONFIG_SMP）都会包含这个函数的代码。

就这句wfc->ret = wfc->fn(wfc->arg);

在ida里是这样的

在这题里我们

第一次使用

a1+32(wfc->fn)=prepare_kernel_cred

a1+40(wfc->arg)=Null

就会执行prepare_kernel_cred(Null)

返回值存到a1+48(wfc->ret)

第二次使用

a1+32(wfc->fn)=commit_creds

a1+40(wfc->arg)=第一次的返回值

就会执行commit_creds(prepare_kernel_cred(Null))提权到root

解法转变为劫持tty_ops->ioctl为work_for_cpu_fn

因为 tty_struct[4] 处成员 ldisc_sem 为信号量，在执行到 work_for_cpu_fn 之前该值会被更改

然后执行两次，一次prepare_kernel_cred(Null)，一次commit_creds(第一次的返回值)

Exp

然而我自己实测下来，依旧不太稳定，仍会kernel panic，或提权失败
